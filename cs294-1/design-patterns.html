Andrew Begel

CS294-1 Reviews

April 2, 2002

 

Design Patterns, Chapter 1 (Gamma, et. al.)

 

I wholeheartedly believe in design patterns. Well, I believe that they’re used by programmers, found in programs, and contribute positively to maintainable code. However, I do believe that, as Gamma alludes to in a very short statement on page 46, the use of certain design patterns (Delegation, Abstract Factory, Observer) makes a lot of code quite a bit harder to understand. As long as someone has a grasp of the patterns used, there is an implicit claim made that the resulting code will be understandable. I do believe that the lack of concreteness (due to abstracting away actual class names, or actual method names, or even actual instances) is cognitively challenging and can get in the way. Perhaps, if someone wrote a tool that could “undo” a design pattern, i.e. reduce the generality to a specific case, in order to show the developer an example of how this code is used at runtime, in a specific case, it would go a long way to convincing me that the imposition of generality for maintenance’s sake is worth it when compared to the increase on time for programmer comprehension.

 

Tool Support for Object-oriented Patterns  (Florijn, Meijers,  van Winsen)

 

This paper I quite liked. It provides a concrete tool for visualizing, creating, editing, and validating design patterns in Smalltalk. Construction is straightforward, but editing generated code is a mess without enough semantic analysis in the toolkit. First, they have the benefit of only needing to work with classes and methods, and not code itself. They use a graph structure for visualizing how the pieces of a pattern fit together (nodes for pieces of pattern, edges for pattern relations), which is itself a composable data structure, both programmatically and cognitively. It also lends itself to a focus+zoom interface where a user could look at an individual design pattern and its fringes without being forced to visualize the entire program. I liked their use of existential logic to declaratively describe a valid pattern; it lends itself to a fairly obvious verification algorithm to validate the correct use of a pattern in the code written. When there is the automatic repair of a broken design pattern, is a preview of the repair made available to the user to verify the repair before it’s made, or does it just happen and you have to figure out after the fact what happened? Is it possible to use a tool such as this pedagogically to evaluate students’ code designs and use of design patterns and steer them in “good” directions (for suitable problem-specific definitions of “good”)? Another question that this paper brings to mind is how widespread the use of patterns are within a given body of object-oriented code? How much is organized via patterns and how much is haphazard? If you used a tool like this for reverse engineering, how much would you really get out of it? I’d like to see a case study using this tool to evaluate a large code base (one that wasn’t designed with the tool) and see whether or not the design patterns are pervasive and help you to understand the architecture and design of the system.
