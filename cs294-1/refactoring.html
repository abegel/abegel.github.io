CS294-1 Reading Review

April 10, 2002

 

Refactoring: Improving the Design of Existing Code (Chapter 2)

 

I understand what refactorings are, and I understand how they’re used, but this chapter feels the need to convince me that refactorings should be used if you’ve got them. Of course, they don’t exist in most programming environments, especially those that I use. I’ve also only seen them for Java and Smalltalk, and other well-behaved object-oriented languages, leaving my C and C++ coding without refactoring support. I do believe that I perform refactoring operations on my codebase quite often, when I’m adding new features, so it frustrates me to no end that I have to resort to refactoring via text editing instead of using a higher-level tool that understands my language syntax and semantics. The philosophy of refactoring also resonates with the division between programming in the small and programming in the large. When you’re programming a small project, indirection, abstraction and modularity just get in the way and cause the project to become larger than necessary. However, when new features come in, refactoring is necessary to abstract a piece here, modularize a piece there, etc. in order to fit the new feature into the current design framework without rewriting it from scratch. In summary, I didn’t need this chapter to convince me that I should use refactorings, however, it might be nice for some chapter to describe how to implement refactorings without specialized tools for all of us working without them.
